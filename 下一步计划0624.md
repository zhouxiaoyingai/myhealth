我对照了当前 v2 需求文档、MVP 计划和开发日志后，判断现在项目已经从“本地 MVP 占位图”进入了“真实 AI 配图可
  用”的阶段，下一步不应该马上扑到 Supabase，而应该先把今日卡片这个核心闭环打稳。

  当前状态和文档有几个关键偏差：

  1. 需求文档还写着“饮食卡片含早/午/晚/加餐”，但你刚刚明确要的是“一日三餐”，代码也已经改成早餐/午餐/晚餐三
     项。
     需要同步更新 /E:/myhealth/2026-06-23-health-assistant-v2-requirements.md，否则后面验收会反复冲突。

  2. 开发日志还写着“Doubao/Seedream integrations remain future work”，但现在豆包 Seedream 已经接入并跑通。
     需要更新 /E:/myhealth/DEVELOPMENT_LOG.md，不然后续接手的人会误判项目状态。

  3. FR-10 “保存到手机 html2canvas → PNG” 是必做，但当前 /E:/myhealth/src/app/page.tsx 里还是：

     window.alert('本地 MVP：请使用浏览器截图保存卡片，后续将接入 html2canvas。');

     这是目前最明显的未完成核心功能。

  4. 豆包返回的图片 URL 是带 X-Tos-Expires=86400 的临时 URL。现在把 URL 存在 localStorage 里，明天可能就失
     效。
     这对“历史记录回看”和“每日建议缓存”是一个真实风险。短期可以接受重新生成，长期要么下载成 base64/blob 存本
     地，要么接 Supabase Storage。

  我建议接下来的优先级是：

  第一优先级：补齐今日卡片闭环
  这一步最值得先做，因为它直接影响用户眼前体验，也对应需求文档 FR-05 到 FR-12、FR-60 到 FR-64。

  具体包括：

  - 把“保存卡片”从 alert 改成真正导出 PNG。
  - 处理远程豆包图片的跨域导出问题。普通 html2canvas 很可能因为远程图片 CORS 导致导出失败，所以更稳的做法是：
      - 服务端新增 /api/image-proxy?url=...
      - 前端展示或导出时走同源代理图片
      - 再用 html2canvas 导出整张卡片

  - 给图片加明确的“AI 生成 · 仅作示意”水印，满足 FR-64。
  - 对豆包生成状态做更清晰的 UI：生成中、失败降级、重新生成成功，不只是一行 fallback 文案。
  - 把饮食图和运动图的 prompt 测试继续保留，避免后面又退化成裁切、不完整、不是三餐图。

  第二优先级：修正文档和验收口径
  这一步不大，但很重要。现在需求文档、开发日志、README 和代码状态已经不完全一致。

  建议马上同步：

  - 需求文档里的饮食卡片从“早/午/晚/加餐”改为“早/午/晚”。
  - 开发日志改成“豆包 Seedream 已接入，当前剩余问题是图片持久化/保存卡片/Supabase”。
  - README 增加说明：豆包图片 URL 是临时 URL，历史持久化后续要用 Storage 或本地缓存。
  - MVP 计划里“Doubao unavailable/use SVG”改成“Doubao available/SVG fallback retained”。

  这一步的价值是防止后面开发方向摇摆。

  第三优先级：本地数据层升级
  计划文档里原本写的是 IndexedDB + localStorage fallback，但当前 /E:/myhealth/src/lib/store.ts 实际是
  localStorage。

  短期 localStorage 能跑，但如果要存：

  - 历史建议
  - 图片 blob/base64
  - 30 天打卡
  - 导出的卡片素材

  localStorage 很快不够稳。建议下一步把 repository 接口保留，底层升级为 IndexedDB。这样后面接 Supabase 也更
  顺。

  这一项可以和“豆包图片持久化”一起做：图片 URL 可以先转 blob 存 IndexedDB，避免 24 小时后失效。

  第四优先级：完善历史页的真实回看价值
  需求里的历史页不是只看打卡曲线，还要求“点击某天 → 当日建议 + 实际打卡对比”。当前实现已经有基础历史页面，但
  下一步应该让它真正对用户有用：

  - 点击某天显示当天建议的三餐和运动。
  - 对比当天实际打卡：吃了什么、做了什么、体重、心情。
  - 显示“建议 vs 实际”的简短差异，比如“午餐未记录”“运动完成 40 分钟”。
  - 保留最近 30 天图片，但要先解决图片 URL 过期问题。

  第五优先级：再进入 Supabase/Auth
  v2 文档里 Supabase 是大块需求：profiles、daily_advices、daily_logs、Auth、RLS、云同步。它很重要，但现在不应
  该作为下一步第一件事。

  原因很简单：当前核心用户体验还缺“保存卡片”和“图片持久化”。如果现在先接 Supabase，会把问题扩大到账号、RLS、
  同步冲突、部署环境变量，但用户眼前仍然不能保存一张完整卡片。

  更合理的进入顺序是：

  1. 本地闭环完成。
  2. 数据模型稳定。
  3. 再把 repository 从 local/IndexedDB 换成 Supabase adapter。
  4. 最后做 Auth 和多端同步。

  我建议马上做的下一步
  最务实的下一步是：实现“保存卡片为 PNG”，同时增加图片代理，保证豆包远程图能被导出。

  这是一个边界清楚、用户马上能感知、也能补齐需求 FR-10/FR-64 的任务。完成后，今日卡片就从“能看”变成“能生成、
  能展示、能保存、失败能降级”的完整闭环。

  2. 调整 Illustration，保存场景下使用代理 URL。
  3. 接入 html2canvas 或等效 DOM-to-image 方案，保存首页两张卡片或整张今日建议。
  4. 更新 README、DEVELOPMENT_LOG、需求文档，把“一日三餐”和“豆包已接入”同步进去。

  这比马上接 Supabase 更符合当前项目状态。当前真正需要打磨的是“AI 生成卡片”这条主路径，而不是先扩展云端复杂
  度
