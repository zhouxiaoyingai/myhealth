# 2026-06-30 项目开发情况总结

## 一、当前项目定位

MyHealth 当前已经不是最初的“本地静态 MVP”，而是一个本地优先、已接入 Supabase 云端同步能力的 AI 健康生活方式助手 MVP。

当前产品核心路径是：

1. 用户填写健康档案。
2. 系统生成当天饮食、运动建议和健康指标。
3. 调用豆包 Seedream 生成饮食 / 运动配图，失败时回退到本地 SVG 插图。
4. 匿名用户数据保存在浏览器 localStorage。
5. 登录用户数据同步到 Supabase Postgres，AI 配图转存 Supabase Storage。
6. 用户可以记录饮食、运动、体重、心情，并在历史页查看最近记录。

健康边界已经明确：只提供生活方式参考，不做医疗诊断。

## 二、代码与架构现状

### 1. 技术栈

- Next.js App Router，当前构建输出显示 Next.js 16.2.9。
- React + TypeScript。
- Tailwind CSS。
- react-hook-form + zod 用于档案表单校验。
- Vitest 做单元测试。
- Playwright 做路由 smoke e2e 测试。
- Supabase Auth / Postgres / Storage 用于登录用户的云端数据和图片持久化。
- html2canvas 用于卡片导出 PNG。

### 2. 页面范围

当前已有页面：

- `/`：今日建议、饮食卡片、运动卡片、BMR/TDEE/目标热量/BMI/宏量指标、重新生成、保存卡片、提醒和免责声明。
- `/profile`：8 字段健康档案表单。
- `/log`：饮食、运动、体重、心情打卡，支持近期补录。
- `/history`：最近 30 天日历、体重趋势、心情趋势、连续打卡、勋章、选中日期复盘。
- `/settings`：主题、账号状态、数据导出、数据清除。
- `/auth`：Supabase 邮箱魔法链接登录。
- `/auth/callback`：Supabase 登录回调。

### 3. 数据层

仓库抽象已经落地在 `src/lib/repo/`：

- `Repository`：统一应用层数据接口。
- `LocalRepository`：匿名模式下使用 localStorage。
- `SupabaseRepository`：登录模式下使用 Supabase Postgres。
- `useRepository()`：根据 `AuthProvider` 登录状态自动切换本地 / 云端实现。
- `migrateLocalToSupabase()`：支持首次登录后的本地数据迁移。

当前本地数据仍是 localStorage，不是 IndexedDB。代码里使用的 key 是：

```text
myhealth.local.v1
```

### 4. Supabase 接入

Supabase 已经从“后续计划”推进到实际实现：

- 已有 `supabase/migrations/0001_init.sql`。
- 建表范围包括：
  - `profiles`
  - `daily_advices`
  - `daily_logs`
- 已启用 RLS。
- 表策略是 owner-only 读写。
- Storage bucket 为 `advices`，private。
- Storage 路径约定为：

```text

```

相关代码已经存在：

- `src/lib/supabase/client.ts`
- `src/lib/supabase/server.ts`
- `src/lib/supabase/admin.ts`
- `src/lib/supabase/storage.ts`
- `src/app/api/image-storage/route.ts`
- `src/lib/supabase/AuthProvider.tsx`

### 5. AI 配图与图片路线

豆包 Seedream 已经接入：

- `/api/advise` 生成建议后调用 `attachDoubaoImages`。
- 匿名用户保留豆包临时 URL 或 SVG fallback。
- 登录用户可把豆包图片上传到 Supabase Storage，并返回 `dietKey` / `exerciseKey`。
- `/api/image-storage` 用 storage key 换 1 小时 signed URL，并校验 key 中的 userId 是否等于当前登录用户。
- `/api/image-proxy` 只代理火山 / 豆包相关白名单域名，用于绕过 html2canvas 的跨域画布污染问题。
- `Illustration` 组件已支持：
  - 远程图片展示；
  - 同源代理 URL；
  - “AI 生成 · 仅作示意”水印；
  - 生成中状态；
  - fallback 提示；
  - 重新生成按钮。

## 六、当前发现的关键问题

### P0：保存卡片功能存在明显实现断点

`src/app/page.tsx` 中 `saveCard` 使用的是父组件 `Home` 内的 `cardRef`：

```tsx
const cardRef = useRef<HTMLDivElement>(null);
```

但实际挂到 DOM 上的 ref 是 `HomeBody` 内部重新创建的另一个 `cardRef`：

```tsx
const cardRef = useRef<HTMLDivElement>(null);
...
<div className="grid gap-6 lg:grid-cols-[1.2fr_.8fr]" ref={cardRef}>
```

父组件的 `cardRef.current` 很可能一直是 `null`，因此点击“保存卡片”会直接 return，不会真正导出图片。

这和 FR-10 “保存卡片为 PNG”的核心验收直接相关，应作为下一步第一优先级修复。

### P0：lint 当前失败

即使生产构建通过，`npm run lint` 当前失败。仓库如果把 lint 作为 CI gate，会阻塞合并或发布。

优先修复：

- `saveCard` 的 `useCallback` 依赖问题。
- 未使用 import / 参数。
- 未使用 eslint-disable 注释。

### P1：E2E 命令超时，未干净退出

Playwright smoke 用例显示都通过，但命令最终 timeout。

当前配置中：

```ts
webServer: {
  command: 'npm run dev',
  url: 'http://127.0.0.1:3000',
  reuseExistingServer: true,
  timeout: 120000
}
```

同时 Next dev server 提示需要配置：

```ts
allowedDevOrigins: ['127.0.0.1']
```

需要确认是 dev server HMR 警告导致等待，还是已有 dev server 复用策略导致进程不能退出。

### P1：文档有历史状态残留

`DEVELOPMENT_LOG.md` 前面仍然保留了较早判断：


但后续 2026-06-24 记录又说明 Supabase 和豆包都已经接入。虽然后文是最新事实，但文档顶部“Current Blocker / Next Step”容易误导接手者。

建议重写 `DEVELOPMENT_LOG.md` 的顶部状态，让它和当前代码保持一致。

### P1：本地数据层仍是 localStorage

计划文档早期写过 IndexedDB，但当前实际实现是 localStorage。

考虑到未来要保存：

- 历史建议；
- 图片 blob / base64；
- 30 天以上日志；
- 导出卡片素材；

localStorage 容量和稳定性都不是长期方案。Supabase 登录用户已经能解决云端持久化，但匿名用户如果继续依赖 localStorage，图片和较大历史数据仍有风险。

- 心情趋势；

这属于用户价值增强项，不影响基础运行，但影响历史页的实际可用性。
